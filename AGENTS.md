# AGENTS.md

Guidance for AI agents working in this repository.

## What this is

High-bandwidth 3-phase **FOC (Field-Oriented Control) motor controller firmware** for
brushless (BLDC/PMSM) actuators, originally written by Ben Katz. It targets the
**STM32F446RE** MCU on the [bgkatz/3phase_integrated](https://github.com/bgkatz/3phase_integrated)
motor-controller board (DRV8323 gate driver, SPI absolute encoder, CAN bus). It aims to be
drop-in compatible with the older mbed-based firmware for the same hardware.

This is bare-metal C firmware built on **STM32 HAL**, generated and maintained through
**STM32CubeIDE** (project described by `motorcontrol.ioc`). It is *not* a desktop/server
program — it cross-compiles to an ELF/BIN that gets flashed to the board.

## Build / flash

Two supported workflows. **Primary: STM32CubeIDE** (GUI, also owns the `.ioc`).
**CLI workflow** (Ubuntu, ST-Link) is documented below and is the day-to-day path here.

- **STM32CubeIDE.** Open the project (`.project` / `.cproject`) and build. Pin assignments,
  clocks, and peripherals are configured in `motorcontrol.ioc`; regenerating code from the `.ioc`
  overwrites peripheral init files (see "Generated vs. hand-written" below).
- **Command-line build** (needs `arm-none-eabi-gcc` + `make`):
  ```
  make -j$(nproc) -C Debug      # or -C Release
  ```
  Artifacts land in `Debug/` or `Release/` (`motorcontrol.elf`, `.bin`, `.map`, `.list`).
  These makefiles are auto-generated by CubeIDE ("Do not edit") but invoke `arm-none-eabi-gcc`
  directly, so they work standalone.
- **Flash** via ST-Link (CubeIDE, `STM32_Programmer_CLI`, or `openocd`) with `motorcontrol.bin`
  at flash origin `0x08000000`.
- There are **no automated tests** — this is real-time motor firmware. Validate changes on
  hardware (a spinning motor) or via the serial menu / CAN interface.

### CLI workflow (Ubuntu + ST-Link)

One-time setup:

```
# Toolchain + flash/debug tools
sudo apt-get install -y gcc-arm-none-eabi binutils-arm-none-eabi gdb-multiarch openocd stlink-tools
# udev rules so flashing works without sudo (then replug the ST-Link)
sudo cp tools/49-stlink.rules /etc/udev/rules.d/
sudo udevadm control --reload-rules && sudo udevadm trigger
```

Target chip: **STM32F446RE** (Cortex-M4F, hard-float `fpv4-sp-d16`). The serial menu (`fsm.c`)
is on **USART2 @ 115200 8N1**.

Daily loop:

| Action | Command |
|--------|---------|
| Build | `make -j$(nproc) -C Debug` |
| Flash | `openocd -f interface/stlink.cfg -f target/stm32f4x.cfg -c "program Debug/motorcontrol.elf verify reset exit"` |
| Build + flash + serial | `./tools/flash.sh -s` |
| Serial menu only | `screen /dev/ttyACM0 115200` (Ctrl-A K to quit) |
| Step-debug | OpenOCD GDB server + `gdb-multiarch Debug/motorcontrol.elf` |

Editor config lives in `.vscode/` (Cortex-Debug + C/C++ extensions): `tasks.json`
(build/clean/flash), `launch.json` (F5 = launch, "Attach" = inspect a running target),
`settings.json` (IntelliSense include paths). Helper scripts and udev rules live in `tools/`.

**CLI build gotchas** (the generated `Debug/`/`Release/` build files were committed from a
Windows CubeIDE install — these fixes are already applied in this repo, noted here so they're
not "fixed" back):

- **Use `make ... all`, not bare `make`.** With these makefiles the default goal resolves to a
  single object; `all` is the real build-everything target.
- **Hardcoded linker-script path.** `Debug/makefile` / `Release/makefile` referenced
  `C:\Users\ben\...\STM32F446RETX_FLASH.ld`; `make` parses the `C:` as a second target
  ("multiple target patterns"). Changed to the repo-relative `../STM32F446RETX_FLASH.ld`.
- **`make -C Debug clean` is destructive.** The CubeIDE clean target is `rm -rf *` *inside*
  `Debug/`, which also deletes the committed `makefile`, `*.mk`, and `objects.list` (the linker
  response file) — not just outputs. CubeIDE regenerates those; the CLI cannot. Use the safe
  clean in `.vscode/tasks.json` (removes `*.o/*.d/*.su` + the elf/bin/map/list outputs only).
  If you do nuke them, `git checkout -- Debug/` restores them.
- **`sysmem.c` newlib fix.** The 2018-vintage `_sbrk` used `caddr_t` without including
  `<sys/types.h>`; newer newlib (arm-gcc 13+) no longer pulls it in transitively. Added the
  include.
- **Build artifacts are committed.** `Debug/`/`Release/` `*.o/*.elf/*.bin` are tracked in git, so
  a rebuild shows up as a large diff. Consider `.gitignore`-ing them (keeping `makefile`, `*.mk`,
  `objects.list`, linker scripts) if you want clean diffs.

## Source layout

Everything meaningful lives under `Core/`:

- `Core/Src/`, `Core/Inc/` — application source and headers.
- `Drivers/` — vendor STM32F4xx HAL + CMSIS. **Don't edit**; treat as a library.
- `Core/Startup/startup_stm32f446retx.s` — reset/vector table.
- `STM32F446RETX_FLASH.ld` / `_RAM.ld` — linker scripts (512K flash, 128K RAM).
- `Debug/`, `Release/` — generated build output; not source.

### Key application modules

| File | Responsibility |
|------|----------------|
| `main.c` | Boot: HAL/peripheral init, load flash prefs, configure DRV8323, start ADC/PWM/CAN, then idle in a `while(1)` printing DRV faults. **Real work happens in interrupts.** |
| `stm32f4xx_it.c` | **The real main loop.** `TIM1_UP_TIM10_IRQHandler` is the 40 kHz commutation ISR: sample ADC → sample encoder → `run_fsm` → service CAN. Also holds the CAN-RX path and UART-RX (serial menu char input). |
| `fsm.c` / `fsm.h` | Finite state machine: `MENU / CALIBRATION / MOTOR / SETUP / ENCODER` modes. Handles the serial terminal menu and setup-parameter editing. |
| `foc.c` / `foc.h` | The control core: dq0/inverse-dq0 transforms, SVM, PI current loop (`commutate`), `torque_control` (the position-velocity-torque impedance law), field weakening, ADC current scaling, offset zeroing. `ControllerStruct` is the central state object. |
| `position_sensor.c` | SPI absolute-encoder read, multi-turn tracking, velocity estimate, electrical-angle computation with linearization LUT. `EncoderStruct`. |
| `calibration.c` | Encoder ↔ rotor calibration: phase-order detection and offset/linearization LUT generation. `CalStruct`. |
| `can.c` | CAN init + the packed command/reply protocol (`pack_reply` / `unpack_cmd`). See protocol notes below. |
| `drv8323.c` | DRV8323 gate-driver SPI register config + fault readout. |
| `preference_writer.c` / `flash_writer.c` | Persist config to a flash sector (sector 6) and reload it. |
| `math_ops.c`, `lookup.h` | Fast math: `sin_lut`/`cos_lut`, fast min/max, `float_to_uint`/`uint_to_float` for CAN packing, `limit_norm`. |
| `hw_config.h` | **Compile-time hardware config** — pin/peripheral handle mapping, scaling constants, current-loop gains, duty-cycle limits, `V_BUS_MAX`. Edit here to port hardware. |
| `user_config.h` | **Runtime (flash-stored) config** — maps named params (`I_MAX`, `CAN_ID`, `KT`, `M_ZERO`, encoder LUT, etc.) onto `__float_reg[]` / `__int_reg[]` flash registers. |

## Architecture / control flow

- **Interrupt-driven, ~40 kHz** (`DT = 25 µs`, `hw_config.h`). `main()` only sets things up; the
  TIM1 update ISR drives everything time-critical. Treat the ISR path as hard-real-time — keep it
  fast, no blocking, no `printf` in the hot path.
- **ISR priorities:** commutation (PWM/TIM1) is priority 0 and preempts CAN. CAN is 1.
- **State machine** (`run_fsm`, called every ISR tick). In `MOTOR_MODE` each tick runs
  `torque_control` → `field_weaken` → `commutate`. Mode changes arrive over serial (single chars,
  see `fsm.h` command defines) or special CAN frames.
- **Control law:** `torque_control` computes a desired torque from the received impedance command
  `t_des = kp*(p_des - θ) + kd*(v_des - ω) + t_ff`, converts to `i_q_des` via `KT*GR`, and
  `commutate` runs the dq-frame PI current loop and space-vector modulation onto TIM1 PWM.

### CAN protocol (host ↔ controller)

- Filter accepts frames addressed to this node's `CAN_ID`; replies are sent to `CAN_MASTER`.
- **Command frame (8 bytes):** packed `p_des, v_des, kp, kd, t_ff` — see the bit layout comment
  above `unpack_cmd` in `can.c`.
- **Reply frame:** packed `id, position, velocity, current/torque, v_bus` — see `pack_reply`.
- **Special command frames** (8 bytes of `0xFF…` with a distinct last byte) enter/exit motor mode
  and zero position: `0xFC` = motor mode, `0xFD` = menu/idle, `0xFE` = set zero. Handled in
  `stm32f4xx_it.c`.
- A CAN watchdog (`CAN_TIMEOUT`) zeros commands if the host stops talking.

## Conventions & gotchas

- **Generated vs. hand-written code.** Files like `main.c`, `can.c`, `adc.c`, `tim.c`, `spi.c`,
  `gpio.c`, `stm32f4xx_it.c` are CubeMX-generated. **Only edit inside the
  `/* USER CODE BEGIN x */ … /* USER CODE END x */` guards** — code outside those markers is
  overwritten if anyone regenerates from `motorcontrol.ioc`. All the project's custom logic lives
  inside those blocks (e.g. the entire ISR body, the CAN protocol in `USER CODE BEGIN 1`).
- **`hw_config.h` = hardware/compile-time, `user_config.h` = flash/runtime.** Don't confuse the
  two. `user_config.h` params are `#define`s onto `__float_reg[]`/`__int_reg[]` and only take
  effect after a power cycle (they're loaded from flash at boot). The setup menu even prints
  "VALUES NOT ACTIVE UNTIL POWER CYCLE".
- **Flash persistence** uses sector 6; `preference_writer_*` open/flush/close/load. Calibration
  and the setup menu write here. Boot sanitizes empty/`-1`/`NaN` flash to sane defaults (see the
  block after `preference_writer_load` in `main.c`).
- **Phase order & sign.** `PHASE_ORDER` swaps phases A/B in both `set_dtc` and `analog_sample` so
  current/torque sign matches encoder direction. If you touch commutation, keep both paths
  consistent. (Recent history shows real bugs here — see git log.)
- **Current scaling depends on `I_MAX`:** the DRV CSA gain switches 40X→20X above 40 A, and
  `i_scale` doubles to match. Keep ADC offset zeroing (`zero_current`) and gain selection aligned.
- **Real-time discipline:** no dynamic allocation in the control path (calibration LUTs are
  statically sized via `PPAIRS_MAX`/`N_CAL`), no floating blocking calls in the ISR, prefer the
  `fast_*` math and LUTs already provided.
- **Units:** angles in radians (mechanical and electrical), velocity rad/s, current A, torque N·m;
  gear ratio `GR` and torque constant `KT` convert motor ↔ output.

## When making changes

- Identify whether the file is CubeMX-generated; if so, stay inside USER CODE guards.
- Touching the control loop, commutation, current scaling, or CAN packing affects a physical motor
  — be conservative, preserve existing limits (`I_MAX`, `V_BUS_MAX`, duty-cycle clamps, voltage
  saturation), and explain any change to safety limits.
- Don't reformat generated files or vendor `Drivers/` code.
- Don't commit `Debug/`/`Release/` build artifacts as if they were source.
